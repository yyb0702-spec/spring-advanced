# Lv6 개선 과제

필수 기능 구현 이후 전체 코드를 다시 점검하며 유지보수성과 일관성 측면에서 개선할 수 있는 부분들을 정리했다.

검토 대상에는 
Validation 응답 구조 차이, 
회원가입 비밀번호 정책 불일치,

가 있었다.

그중 이번 리팩토링에서는 다음 두 가지를 우선 개선 대상으로 선정했다.

- Validation 실패 시 응답 형식이 기존 커스텀 예외 응답과 다르게 반환되는 문제
- 회원가입 시 비밀번호 정책 검증이 누락된 문제

선정 기준은 다음과 같았다.

- 실제 API 응답 일관성에 영향을 주는가
- 코드 의도가 명확하지 않은 부분인가
- Spring/JPA의 권장 방식과 어긋나는가
- 변경 전후 차이를 명확하게 설명할 수 있는가
---
## 문제 분석
###   문제 1. Validation 예외 응답 구조 불일치

프로젝트에서는 커스텀 예외를 GlobalExceptionHandler에서 공통 형식으로 처리하고 있었다.
```
{
"status": "BAD_REQUEST",
"code": 400,
"message": "에러 메시지"
}
```
하지만 @Valid 검증 실패로 발생하는 MethodArgumentNotValidException은 별도 처리하지 않아 

Spring 기본 에러 응답이 반환될 수 있었다.

즉 동일한 400 Bad Request 상황이어도 예외 종류에 따라 응답 구조가 달라지는 문제가 존재했다.

이 문제는 API 사용자 입장에서 응답 포맷 예측을 어렵게 만들고, 

프론트엔드에서 예외 처리를 일관되게 구현하기 어렵게 만든다.

### 문제 2. 회원가입 비밀번호 정책 불일치

비밀번호 변경 요청 DTO에는 다음 정책이 적용되어 있었다.

8자 이상
숫자 포함
대문자 포함

하지만 회원가입 DTO에서는 단순 @NotBlank만 적용되어 있었다.

그 결과 회원가입 시에는 약한 비밀번호로 가입할 수 있지만, 

이후 비밀번호 변경 시점에는 강한 정책을 요구하는 비일관적인 흐름이 발생할 수 있었다.

비밀번호 정책은 사용자 계정 생성 시점부터 동일하게 적용되는 것이 자연스럽다고 판단했다.

---
## 해결 과정
### Validation 예외 응답 형식 통일
GlobalExceptionHandler에 MethodArgumentNotValidException 처리 메서드를 추가했다. 
   
검증 실패 시 첫 번째 필드 에러 메시지를 꺼내 기존 getErrorResponse() 메서드를 통해
동일한 응답 구조로 반환하도록 했다.

이를 통해 커스텀 예외와 Bean Validation 예외가 같은 응답 형식을 사용하게 되었다.

### 회원가입 비밀번호 검증 추가
   SignupRequest의 password 필드에 @Pattern을 추가했다.

검증 규칙은 비밀번호 변경 요청과 동일하게 설정했다.

```
8자 이상
숫자 포함
대문자 포함
```

이제 회원가입과 비밀번호 변경 모두 동일한 비밀번호 정책을 따른다.

---

## 개선 후 효과
- 잘못된 요청에 대한 에러 응답 형식이 더 일관되게 유지된다.
- 회원가입과 비밀번호 변경의 비밀번호 검증 정책이 동일해졌다.
- 작은 수정이지만 검증, 예외 처리, 엔티티 매핑의 의미가 더 분명해졌다.
---
## 회고
Validation을 적용하더라도 예외 처리 방식까지 함께 고려해야 실제 API 사용성이 좋아진다는 점을 배울 수 있었다. 

서버 입장에서는 단순한 검증 실패일 수 있지만, 클라이언트 입장에서는 응답 구조가 매번 달라지면 처리 비용이 커질 수 있기 때문이다.

비밀번호 정책 역시 특정 기능에만 적용하는 것이 아니라, 서비스 전체 기준에서 일관되게 관리해야 한다는 점을 확인할 수 있었다. 

이번 개선을 통해 DTO Validation, 전역 예외 처리, 서비스 로직의 역할을 어떻게 분리해야 하는지 조금 더 명확하게 이해할 수 있었다.
